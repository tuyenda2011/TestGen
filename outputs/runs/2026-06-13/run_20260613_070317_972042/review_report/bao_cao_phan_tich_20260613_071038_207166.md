## Test còn thiếu
- **Kiểm tra thành công của `deactivate()`** khi số dư bằng 0 (đảm bảo tài khoản có thể được hủy mà không ném ngoại lệ).
- **Kiểm tra `withdraw` khi số dư bằng 0** và rút 0 (đảm bảo hành vi hợp lệ hoặc ném ngoại lệ phù hợp).
- **Kiểm tra `deposit` với số tiền bằng 0** (đảm bảo hành vi đúng theo yêu cầu `amount <= 0` → ném `IllegalArgumentException`).
- **Kiểm tra `transfer` khi số tiền lớn hơn số dư nguồn** (đảm bảo ném `IllegalStateException` từ `withdraw`).
- **Kiểm tra `transfer` khi nguồn và đích là cùng một tài khoản** (trường hợp biên chưa được xác định trong source).

## Rủi ro maintainability
- Các giá trị “magic number” (ví dụ `1000.0`, `0.0`) được hard‑code trong nhiều test; nếu thay đổi yêu cầu sẽ phải cập nhật nhiều nơi.
- Các test phụ thuộc vào trạng thái mặc định `isActive = true` được thiết lập trong constructor; nếu logic khởi tạo thay đổi, nhiều test sẽ phá vỡ đồng thời.
- Không có comment mô tả rõ ràng mục đích của một số test (như TC‑023) – có thể gây khó hiểu khi bảo trì.

## Gợi ý sửa ngay
1. **Thêm test cho `deactivate()` thành công**  
   ```java
   @Test
   @DisplayName("TC-024: Verify deactivate succeeds when balance is zero")
   void testDeactivateSuccess() {
       BankAccount account = new BankAccount("ACC123", 0.0);
       assertDoesNotThrow(account::deactivate);
       assertFalse(account.isActive());
   }
   ```

2. **Thêm test cho `deposit` với số tiền 0**  
   ```java
   @Test
   @DisplayName("TC-025: Verify deposit throws exception for zero amount")
   void testDepositZeroAmount() {
       BankAccount account = new BankAccount("ACC123", 1000.0);
       assertThrows(IllegalArgumentException.class, () -> account.deposit(0.0));
   }
   ```

3. **Thêm test cho `withdraw` khi số dư bằng 0 và rút 0** (hoặc rút >0) để xác nhận hành vi biên.  
4. **Thêm test cho `transfer` khi số tiền > số dư nguồn** để kiểm chứng ngoại lệ `IllegalStateException`.  
5. **Rà soát và thay thế các hằng số lặp lại** bằng các constant static final trong lớp test để dễ bảo trì.
