## Lỗi nghiêm trọng
- **TC‑002**: Kịch bản mô tả “Create post missing required userId” nhưng script kiểm thử vẫn **mong đợi status 201** và chỉ kiểm tra có `id`. Theo yêu cầu, `userId` là trường bắt buộc; API thực tế (JSONPlaceholder) trả về **status 201** nhưng đây không phải hành vi mong muốn của hệ thống cần kiểm thử. Test này mâu thuẫn với quy tắc kinh doanh và sẽ gây hiểu lầm. Cần sửa để **kiểm tra rằng response không phải 201** (ví dụ 400) hoặc ít nhất xác nhận `userId` không có trong response.
- **TC‑005**: Sử dụng `pm.expect([404, 400]).to.include(pm.response.code);` là cú pháp hợp lệ, nhưng không có **assertion kiểm tra nội dung lỗi** (ví dụ thông báo). Tuy không phải lỗi nghiêm trọng, nhưng thiếu kiểm chứng đầy đủ.

## Test còn thiếu
- Kiểm thử **negative** cho trường `title` hoặc `body` nếu chúng được coi là bắt buộc trong một số quy tắc nghiệp vụ (không có trong yêu cầu hiện tại nhưng thường cần).
- Kiểm thử **boundary** thực tế với chuỗi dài **255 ký tự** và **5000 ký tự** (hiện chỉ dùng placeholder `"MAX_TITLE_255"`/`"MAX_BODY_5000"`). Cần tạo payload thực sự đạt độ dài tối đa để xác nhận API không cắt ngắn hoặc trả về lỗi.
- Kiểm thử **security**: xác nhận API không trả về dữ liệu nhạy cảm khi gửi payload có chứa script (XSS) hoặc JSON không hợp lệ.
- Kiểm thử **exception** khi server trả về lỗi 5xx (giả lập bằng mock) – không thể thực hiện trên API công cộng nhưng nên có kịch bản mô phỏng.

## Assertion yếu
- **TC‑002** chỉ kiểm tra `status 201` và sự tồn tại của `id`; không kiểm chứng rằng `userId` thực sự bị thiếu hoặc giá trị mặc định nào được trả về. Assertion quá yếu.
- **TC‑003** dùng placeholder `"MAX_TITLE_255"`/`"MAX_BODY_5000"` thay vì chuỗi thực tế dài 255 và 5000 ký tự → không thực sự kiểm chứng giới hạn độ dài.
- **TC‑005** chỉ kiểm tra mã lỗi, không kiểm tra nội dung thông báo lỗi hay cấu trúc response.

## Rủi ro maintainability
- **Phụ thuộc vào API công cộng** (`https://jsonplaceholder.typicode.com`). Nếu dịch vụ tạm ngưng hoặc thay đổi hành vi, các test sẽ thất bại (flaky).
- Không có **mock** cho các trường hợp lỗi (400, 404, 5xx); việc chạy trên môi trường thực sẽ gây độ trễ và không kiểm soát được dữ liệu.
- Sử dụng **placeholder** cho dữ liệu dài gây nhầm lẫn khi bảo trì; người đọc có thể nghĩ rằng chuỗi đã đạt độ dài yêu cầu.

## Gợi ý sửa ngay
1. **Sửa TC‑002**  
   ```javascript
   pm.test("Status code is not 201", function () {
       pm.expect(pm.response.code).to.not.eql(201);
   });
   pm.test("Response returns error for missing userId", function () {
       pm.expect(pm.response.code).to.be.oneOf([400, 422]);
   });
   ```
2. **Cập nhật TC‑003** để tạo chuỗi thực tế:
   ```javascript
   var longTitle = "a".repeat(255);
   var longBody  = "b".repeat(5000);
   // dùng longTitle và longBody trong payload
   ```
3. **Thêm test negative** cho `title`/`body` (ví dụ gửi `title` rỗng nếu quy tắc yêu cầu không cho phép).
4. **Thêm test security** gửi payload chứa `<script>alert(1)</script>` và xác nhận response không phản hồi script.
5. **Cân nhắc mock server** (ví dụ dùng `postman-echo` hoặc một server nội bộ) cho các kịch bản lỗi 4xx/5xx để giảm phụ thuộc mạng.
6. **Document rõ ràng** trong mô tả test rằng các test dựa vào API công cộng và có thể bị ảnh hưởng bởi trạng thái mạng.
